# v0.5.0 — Native iOS Companion (In Progress)

The focus of this version is making the iOS app a polished, first-class interface for communicating with agents — covering all three modes (auto voice, PTT, typing) with appropriate system integrations.

## Completed in this milestone

- Three-way input mode picker: Auto / PTT / Typing
- Typing mode: text input, no TTS/STT, hub `text_mode` flag
- PTT mode: hold-to-talk with ZStack-centered mic button, no auto-record interference
- Background voice mode: silence keepalive loop, VAD auto-stop
- Live Activity: mode-aware display with `voicechat://mic` deep link
- Settings: Done button, Connect in Server section
- Hub: `session_ended` no longer kills sessions (fire-and-forget turns)
- Hub: `session_list` cleans up stale iOS-side sessions on reconnect
- Hub: `thinking` signal sent at start of `handle_converse`
- Session timeout: increased to 120 minutes
- Thinking indicator: shown immediately on `thinking` message from hub
- Spotify fix: audio session interruption observer pauses silence loop
- Local notifications: sent on `assistant_text` when app is backgrounded
- Typing mode: no Live Activity (notifications are the primary channel)

## In Progress / Planned for v0.5.0

### Live Activity Revamp

Three distinct Live Activity designs based on input mode:

**Auto mode** — Full Live Activity with:
- Dynamic Island compact: status dot + voice name + waveform/listening icon
- Dynamic Island expanded: voice name, status, last message, mic state indicator
- Lock Screen: voice name, status, last message truncated, "Listening" or "Speaking" pill

**PTT mode** — Compact Live Activity:
- Dynamic Island compact: voice name + mic icon
- Lock Screen: "Open to Talk" button (deep links to app, starts PTT)
- No auto-record indicators (user-initiated only)

**Typing mode** — No Live Activity
- Rely on push notifications for all communication
- Notification actions: "Reply" (opens app to typing view)

### Notifications System

Smart notification strategy per mode:

| Event | Auto | PTT | Typing |
|-------|------|-----|--------|
| Agent responds (background) | Sound only (audio plays) | Alert + sound | Alert + sound |
| Agent ready for input (background) | Auto-records, no notification | Subtle "ready" notification | "Ready for message" notification |
| Session becomes ready | No notification | No notification | "Claude connected" |
| Task completed (`session_ended` received) | Notification if backgrounded | Notification | Notification |

Notification categories:
- `AGENT_RESPONSE`: actions "Open" + "Dismiss"
- `AGENT_TYPING_RESPONSE`: actions "Open to Reply" + "Dismiss"

### Action Button Integration

Register an `AppShortcut` via App Intents so users can assign "Toggle Recording" to the iPhone Action Button (iPhone 15 Pro, 16+):

```swift
struct ToggleRecordingShortcut: AppShortcutsProvider {
    static var appShortcuts: [AppShortcut] {
        AppShortcut(
            intent: ToggleRecordingIntent(),
            phrases: ["Toggle voice chat recording"],
            shortTitle: "Record"
        )
    }
}
```

Users assign via Settings > Action Button > Shortcut > VoiceChat > "Toggle Recording".

**Note:** Apple's PTT Framework (for walkie-talkie style apps) is NOT appropriate here — it requires a special entitlement, APNs infrastructure, and fundamentally different audio session management. The AppShortcut approach is lighter weight and sufficient for our use case.

### Text Mode Evolution

Potential directions (exploring):

1. **Rich message types**: Hub sends tool call results as styled non-spoken messages
   - Protocol: `assistant_text` with `spoken: false` flag = display only, no TTS
   - iOS shows these in a distinct style (monospace, different bubble color)
   - Use case: `bash` output, file edits, search results

2. **Minimal Claude Code wrapper**: In typing mode, render the raw Claude Code output stream
   - Tool use: show tool name + result in collapsible cards
   - File edits: show diff view
   - Would require hub changes to forward raw MCP events

3. **Hybrid mode**: Agent speaks responses but tool calls are shown as text only
   - Most natural for agentic workflows
   - User hears summary, sees detailed output

Decision: start with option 1 (non-spoken messages via protocol extension) as it's the least invasive.
