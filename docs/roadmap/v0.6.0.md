# v0.6.0 — MCP Server Overhaul

Rethink how the MCP server bridges agents and the hub to enable accurate status reporting, message queuing, and inter-agent communication.

## Agent Status Reporting

- [ ] **"Working" state** — After a `converse()` call returns, the hub shows "Working..." instead of "Ready" so the user knows the agent is still active between converse cycles.
- [ ] **`report_status` tool** — New MCP tool that lets the agent tell the hub what it's doing between converse calls (e.g. "Reading files...", "Running tests..."). Hub forwards to browser for display.
- [ ] **Status timeout** — If no `converse()` or `report_status()` call arrives within N seconds, browser shows "Idle" to distinguish from active work.

## Message Queuing

- [ ] **User messages while agent works** — Allow the user to record or type messages while the agent is between converse calls. Messages are queued in the hub and delivered on the next `converse(wait_for_response=true)`.
- [ ] **Queue indicator** — Browser shows a badge or indicator when messages are queued, so the user knows their message was captured.
- [ ] **`check_inbox` tool** — New MCP tool that lets the agent poll for queued user messages without blocking on audio. Agent can check mid-task and decide whether to address the message or continue working.

## Inter-Agent Communication

### Problem

Agents currently have no way to communicate with each other. Each session is completely isolated — the only interaction channel is between an agent and the human via voice. Use cases like "tell Echo to summarize this" or agents collaborating on a task behind the scenes are impossible.

Typing directly into another agent's tmux session doesn't work because the agent is typically blocked in a `converse()` call waiting for voice input. Text sent to the shell prompt sits there unprocessed until the current turn ends, and even then it's treated as a shell command, not Claude input.

### Design Direction

**New MCP tools:**

- [ ] **`list_agents`** — Returns currently active agents with their names and status (ready, active, listening, idle). Agents can call this anytime to discover who's online, since sessions can be spawned or terminated mid-conversation.
- [ ] **`send_message(target, message)`** — Sends a text message to another agent by voice name. The hub stores it in the target agent's inbox (in-memory on the hub, not on disk).
- [ ] **`check_inbox`** — Returns any pending messages immediately (non-blocking). Agents can call this between converse cycles to check for inter-agent messages.

**Message delivery via converse:**

- When an agent calls `converse(wait_for_response=true)`, the hub checks the agent's inbox before/after capturing voice. Pending messages are bundled into the converse result alongside the user's speech (e.g. `"User said: ... | Message from Adam: ..."`).
- If the agent is blocked waiting for voice and a message arrives, the hub can optionally interrupt the wait and return the message immediately, so agents are always reachable.
- This piggyback approach means agents don't need to explicitly poll — messages arrive naturally with each converse cycle.

**Agent autonomy:**

- The receiving agent decides whether to speak the message to the user or handle it silently.
- Agents can reply back to the sender via `send_message`, enabling back-and-forth without human involvement.
- The human can disconnect and agents can continue communicating as long as the hub is running — voice I/O is only needed for human interaction, not inter-agent messaging.

### Open Questions

- Should there be a message priority system (urgent messages interrupt converse, normal messages wait)?
- Should the hub provide a shared scratchpad / key-value store for passing structured data between agents?
- How should agents handle message ordering and conversation threading across multiple agents?
