# v0.6.0 - Agent Orchestration

Dual-backend architecture, agent-to-agent communication, sub-agent workers, and a public API.

## Dual Backend

The hub abstracts its session backend behind a pluggable interface. OpenClaw is the primary path forward. Claude Code via tmux stays as a legacy option.

```
Hub (FastAPI)
└── Session Backend (pluggable)
    ├── OpenClaw Backend ← primary
    └── tmux Backend ← legacy
```

The **OpenClaw backend** connects to a local Gateway over WebSocket. It gives full visibility into the agent loop, native multi-agent support, model-agnostic routing, and block streaming for progressive TTS. The **tmux backend** is the current architecture — Claude Code spawned in tmux sessions, communicating via MCP. It uses a flat-rate subscription and has deep coding capabilities. The frontend doesn't know or care which backend runs underneath.

Both backends implement the same interface: `spawn_session`, `terminate_session`, `send_message`, `get_status`, `list_sessions`, and `on_event`. Migration starts by abstracting this interface, refactoring tmux into one backend, building the OpenClaw backend alongside it, then defaulting new installs to OpenClaw.

## Status Reporting

Right now the hub has no visibility between `converse()` calls. The browser shows "Ready" even while the agent is actively working.

After `converse()` returns, the session transitions to "Working." Agents can push finer-grained updates with a new `report_status("Reading files...")` tool, forwarded to the browser as WebSocket status events. If 60 seconds pass with no `converse()` or `report_status()`, the status flips to "Idle."

Compaction detection is trickier. When Claude Code compacts its context the agent goes silent. The hub can poll tmux output for compaction text, hook into Claude Code lifecycle events, or fall back to a timeout heuristic showing "Compacting..." in the UI.

## Message Queuing

Users can't send messages while an agent is working between converse cycles. The fix: let the browser accept input at any time. If the agent is busy, the hub queues the message and delivers it on the next `converse(wait_for_response=true)` call. A badge on the session tab shows how many messages are queued.

A new `check_inbox` tool gives agents a non-blocking way to poll for pending messages mid-task. Messages also arrive automatically with each converse call, so polling is optional.

## CLI-Based Agent Interface

The MCP server is replaced by `clawmux`, a CLI tool for voice and inter-agent messaging. Moved to [v0.5.0](v0.5.0/cli.md) scope.

## Sub-Agent Orchestration

Each main agent is a **leader** that can spawn **workers** — lightweight sub-agents that inherit the leader's voice and appear nested in the browser sidebar.

```
Sky (leader)
├── Sky-A (refactoring auth)
└── Sky-B (running tests)
```

Leaders get five new tools: `spawn_worker(task)` creates a worker with a task description, `list_workers` checks status, `kill_worker(name)` terminates one, and `message_worker` / `broadcast_workers` handle communication. Workers get the standard tools (`converse`, `report_status`, `send_message`, `check_inbox`) but cannot spawn their own workers. Single-level hierarchy only.

Workers appear indented under their leader in the sidebar. Clicking one opens its conversation with a text input. Leaders get a collapsible "Workers" panel showing all worker status and quick actions. Max 4 workers per leader, 14 total. Idle workers auto-terminate after 30 minutes. Killing a leader cascades to all its workers.

Implementation details live in the [orchestration reference](../agents/reference/orchestration.md).

## Public API

The hub already has a REST + WebSocket API that the browser uses. v0.6.0 formalizes it for external clients — iOS apps, desktop apps, home automation, third-party integrations.

**Authentication** is token-based. The hub generates a token on first run, stored in `data/settings.json`. Clients pass it as a WebSocket query param or REST Bearer token.

**Versioning** prefixes REST endpoints with `/v1/`. WebSocket messages get a `version` field. Unversioned endpoints stay for backward compatibility.

New endpoints cover agent management: listing agents with status and worker counts, sending messages to agents, reading inboxes, and spawning or listing workers. New WebSocket events stream worker lifecycle and agent status in real time.

Full protocol reference: [protocol docs](../agents/reference/protocol.md)
